04 — Architecture

System architecture and technology stack

Modern JAMstack architecture: Next.js App Router on Vercel (frontend + API), Supabase (database + auth + edge functions), with multi-provider AI and Zoom integration.

Technology Stack

LayerTechnologyDetails
Frontend Next.js 16 + React 19 + TypeScript App Router, Server Components, Tailwind CSS, Shadcn/UI, Headless UI
Backend Next.js API Routes + Supabase Edge Functions 47 route handlers, Zod validation, standardized response contract
Database PostgreSQL (Supabase) 25+ tables, Row Level Security, 19 migration files, typed repositories
Auth Supabase Auth (GoTrue) JWT in HTTP-only cookies, role-based access (admin / teacher / client)
AI Gemini + OpenAI + Anthropic Multi-provider abstraction, 26+ tools, AES-256-GCM key encryption
Video Zoom API + Webhooks Meeting creation, student registration, webhook-driven attendance
Hosting Vercel + Supabase Cloud CDN edge delivery, preview deploys on PR, auto-deploy on main
Monitoring Sentry Error tracking (server + client), source maps, session replay
CI/CD GitHub Actions Lint, typecheck, Vitest, coverage, Playwright E2E on every PR

Database Schema

PostgreSQL with 25+ tables organized into three domains:

D

Core Domain (9 tables)

users, companies, students, groups, group_students, periods, class_sessions, attendance, grades

Z

Zoom Integration (4 tables)

zoom_meetings, zoom_registrants, zoom_webhook_events, attendance_logs

O

AI & Operations (12 tables)

ai_providers, ai_conversations, ai_messages, ai_feedback, system_logs, alerts, user_alerts, pipeline_students, teacher_availability, and more

Security Architecture

  • Row Level Security (RLS) on all database tables — admins see everything, teachers see assigned groups, clients see own company
  • JWT authentication via Supabase Auth with secure HTTP-only cookies
  • requireAdmin() guards on all admin API routes (server-side verification)
  • AES-256-GCM encryption for AI provider API keys at rest
  • Rate limiting middleware on all API routes (sliding window)
  • Zod schema validation on all API inputs

06 — Roadmap

12-month evolution plan

Four quarterly phases move the platform from operational tool to intelligent automation engine. Each phase builds on the previous, with clear milestones and measurable outcomes.

Q2 2026 — Foundation & Client Expansion

Current

Theme: Close the client portal gap and eliminate critical tech debt.

  • Expand client portal from 2 to 8+ pages (progress dashboards, enrollment requests, communication, billing visibility)
  • Decompose god components (students page at 2,500+ lines, edit-group drawer at 934 lines)
  • Remove @ts-nocheck from all 50 flagged files (data integrity risk)
  • Implement distributed rate limiting (Upstash/Redis) for serverless scaling
  • Add real-time subscriptions for attendance and notifications via Supabase Realtime

Q3 2026 — Intelligent Automation

Planned

Theme: Make the platform proactively helpful instead of reactive.

  • Automated teacher-group scheduling based on availability, level match, and workload balance
  • Predictive attendance alerts (notify admin/client when patterns suggest dropout risk)
  • Scheduled report generation and delivery (PDF/email on configurable cadence)
  • Smart substitution suggestions when teachers are unavailable

Q4 2026 — Multi-Agent Management

Planned

Theme: AI agents handle routine operations; humans handle exceptions.

  • Agent-driven student pipeline (auto-assess, auto-place, auto-enroll with human approval gates)
  • Natural language operational commands ("Move Maria from Group A to Group B next period")
  • AI-generated period summaries with actionable recommendations
  • Automated Zoom meeting lifecycle management

Q1 2027 — Predictive Analytics & Scale

Planned

Theme: Data-driven school management at scale.

  • Student outcome prediction (at-risk identification based on attendance + grade patterns)
  • Teacher effectiveness analytics (attendance correlation, grade distributions, feedback)
  • Revenue forecasting from pipeline trends, enrollment velocity, and churn signals
  • Multi-school support (single instance managing multiple locations)
